## Web/API/Shared Boundaries

- `apps/web` consumes `apps/api` through `apps/web/lib/api.ts`.
- `packages/shared` owns common constants and Zod schemas used by API and web types.
- API owns all backend state, vault, signing, database writes, scheduler, and Telegram delivery.
- Web surfaces runtime status, warnings, forms, confirmations, empty/error/loading states, and calls API routes.
- No source modification outside `report_md/` was performed during this audit.

## Data Flow

1. Operator authenticates through `/api/auth/login`.
2. Web server/client calls API with session cookie and CSRF header for mutating requests.
3. API validates request body/params with shared Zod schemas and local helpers.
4. API reads/writes Postgres via Drizzle.
5. Redis/BullMQ carries scheduler and notification work.
6. RPC/Basescan links are used for chain status, balances, quote simulation, receipt refresh, and explorer URLs.

## Transaction Flow

1. Wallet/pair/router/token rules are loaded.
2. Dry-run or execute-once validates amount, pair, wallet, risk, quote, and router.
3. Execute-once requires idempotency key and per-wallet lock.
4. Live path requires non-demo, non-dry-run, explicit confirmation, vault unlock, allowance, quote validation, simulation, and signing.
5. Transaction row stores request, hash, status, calldata/quote/simulation hashes, confirmation metadata, and Basescan URL.
6. Confirmation refresh moves rows through submitted, pending finality, finalized, failed, stuck, or dropped.

## Scheduler Flow

1. `scheduler-service.ts` acquires a singleton DB lock.
2. Due enabled wallet schedules enqueue DRY_RUN jobs.
3. Trade worker creates scheduled dry-run records.
4. Confirmation worker tracks submitted live transactions.
5. Notification worker sends/audits Telegram messages.
6. Live scheduled execution is rejected by design.

## Notification Flow

1. API actions call `createTelegramService`.
2. Settings are read from `telegram_settings`.
3. Bot token is decrypted only for send.
4. Message metadata is formatted and sent to Telegram when enabled.
5. `notification_deliveries` records sent, failed, and skipped attempts.

## UI Flow

1. `/login` renders a dark operator login card.
2. Authenticated app routes render `AppShell`.
3. Top bar shows demo/dry-run/live, vault, emergency pause, Base chain, block, RPC, quote provider, and logout.
4. Dashboard summarizes metrics, safety, queue health, risk limits, and scheduler controls.
5. Wallet detail exposes dry-run, execute-once, pair rules, allowances, schedule, and transaction history.
6. Settings expose security/runtime/vault/emergency and Telegram controls.

## Deployment Artifact Flow

- Local: `docker-compose.yml` starts Postgres and Redis.
- API/web dev: root `pnpm dev`, `pnpm demo`, or package-level scripts.
- Production example: `docker-compose.prod.example.yml`, `apps/api/Dockerfile`, `apps/web/Dockerfile`, `infra/nginx/nginx.conf`.
- Backup: `scripts/backup/backup-postgres.sh`, `restore-postgres.sh`, and wallet vault warning doc.
